# Чеклист: устройство и ускорение бота (Flowise AgentFlow V2 + свой виджет)

Документ для **самопроверки** перед релизом и при жалобах на «медленно». Учтены: архитектура из `info/bot.md`, **только AgentFlow V2**, обсуждения по латентности/стримингу/RAG, реалии 2026 (измеряйте этапы, не гадайте).

**Официальные опоры Flowise:** [AgentFlow V2](https://docs.flowiseai.com/using-flowise/agentflowv2) · [Prediction API](https://docs.flowiseai.com/api-reference/prediction) · [Streaming (SSE)](https://docs.flowiseai.com/using-flowise/streaming) · [Prediction (Using Flowise)](https://docs.flowiseai.com/using-flowise/prediction) (в т.ч. `form` для V2).

---

## A. Архитектура и границы ответственности

| # | Проверка | Критично |
|---|----------|----------|
| A1 | Диалог идёт через **Prediction API** (`POST /api/v1/prediction/{id}`), не через «левые» HTTP-ноды к backend заявок | Flowise не должен слать лиды сам по HTTP — контракт `bot.md` |
| A2 | Заявки — только **виджет →** `POST …/lead/send-lead` → **cesi-backend**; CORS + OPTIONS для preflight настроены в nginx | Иначе `Failed to fetch` в браузере |
| A3 | **nginx**: один `server_name` на боевой хост, `/api/*` только на Flowise, статика виджета отдельным location | Дублирующие `server_name` ломают маршрутизацию |
| A4 | **Docker-сеть**: Flowise и векторное хранилище (Chroma/Qdrant) в **одной** сети | Иначе RAG «молча» пустой |
| A5 | Версия Flowise **зафиксирована** (не `latest` без контроля), есть backup `database.sqlite` / экспорт потоков | Воспроизводимость и откат |

## B. Измерения (без этого оптимизация вслепую)

| # | Проверка | Критично |
|---|----------|----------|
| B1 | Есть **разбиение латентности** по этапам: эмбеддинг запроса → retrieval → (rerank) → LLM TTFT → полный ответ → Parse/сериализация | Узкое место чаще LLM и объём контекста, не Parse |
| B2 | **Боевые SLA** меряются с **того же региона/топологии**, что у пользователя (VPS → API модели), а не только с домашнего ПК | Локалка вводит лишний RTT |
| B3 | Логируются **ошибки и таймауты** по prediction; при Parse — полезно логировать рассогласование `tone_llm` / `tone_final` (если внедрено) | Диагностика «модель ленится» vs баг графа |

## C. Граф AgentFlow V2 (логика и «лишняя работа»)

| # | Проверка | Качество vs скорость |
|---|----------|---------------------|
| C1 | **IF / Router** не вызывают RAG, когда: воронка записи (`leadIntent ≠ none`), явный запрет retrieval (`should_retrieve = false`), сценарий «ситуация» по вашим правилам | Меньше шагов = быстрее и дешевле |
| C2 | **История в LLM** усечена разумно (последние N сообщений / токен-бюджет) | Длинная история бьёт и prefill, и стоимость |
| C3 | **Системный промпт** не раздут: каждый блок должен окупаться поведением; дубли с `bot_behaviors` / JSON-контрактом убрать | Меньше входных токенов без потери правил |
| C4 | **Structured output / JSON**: одна схема, Parse не «чинит» то, что модель системно ломает — тогда правьте промпт или схему | Иначе лишние ретраи и handoff |
| C5 | Нет лишних **последовательных** LLM-вызовов «на всякий случай»; если два вызова — осознанный trade-off (качество vs latency) | Два вызова ≈ почти двойное ожидание |

## D. RAG (ретривер, контекст, качество)

| # | Проверка | Критично |
|---|----------|----------|
| D1 | **topK** и размер чанка согласованы: не тащить в промпт целые простыни без выигрыша в ответе | Главный рычаг prefill + длина генерации |
| D2 | **Фильтры метаданных** (тема, подтема, коллекция) до/после векторного поиска — чтобы отсев был дешёвым | Меньше мусора в контексте |
| D3 | Рассмотреть **двухстадийный** пайплайн: широкий retrieve (N чанков) → **rerank** → в LLM только top‑3…5 | В индустрии часто **снижает** end-to-end за счёт меньшего контекста и более короткого ответа; всё равно измерять на своих данных |
| D4 | **Кэш эмбеддингов** повторяющихся/близких формулировок (нормализация текста) — ускоряет повторные запросы | Не путать с кэшем готового ответа |
| D5 | **Семантический кэш ответов** (Redis и т.п.) **до** тяжёлой цепочки — только с политикой: порог similarity, инваладация при обновлении KB, не ломать **состояние воронки** | Иначе «быстро, но неправильно по этапу диалога» |
| D6 | Единый движок **на тесте и на бою** (переход Chroma → Qdrant): одинаковые чанки, метаданные, фильтры | Иначе «на локалке ок, на проде нет» |

## E. Виджет и контракт ответа

| # | Проверка | Критично |
|---|----------|----------|
| E1 | Боевой сайт грузит **`widget.js`** с **боевым** `AGENTFLOW_ID`; тест нового потока — **отдельный** файл (`widget-staging.js`) или параметр ID, **без** смены кода на сайте клиента | Изоляция теста от прода |
| E2 | Сейчас типичный путь: `fetch` + **полный JSON** — **TTFD** (первая буква) = после всей генерации + Parse | Нормально для JSON-контракта; стриминг — отдельная работа |
| E3 | Индикатор «печатает» — **UX**, не стриминг токенов | Не путать при общении с заказчиком |

## F. Стриминг (SSE) — опционально, если цель «ощущение скорости»

| # | Проверка | Источник / риск |
|---|----------|-----------------|
| F1 | В API предусмотрено `"streaming": true` на том же **prediction** ([док](https://docs.flowiseai.com/using-flowise/streaming)) | Ок для эксперимента |
| F2 | **AgentFlow V2** заявляет SSE и обновления исполнения ([док](https://docs.flowiseai.com/using-flowise/agentflowv2)) | На сложных графах проверяйте фактическое поведение нод |
| F3 | **Свой виджет** должен читать **SSE-поток** (`ReadableStream` / парсинг `event:`/`data:`), а не только `res.json()` | Иначе стриминг бесполезен |
| F4 | **Политика событий**: в UI показывать, например, только `token` для пользовательского текста; служебное (`sourceDocuments`, детальный tool-stream) — в лог или отфильтровать | В сообществах поднимали риск «протекания» внутренностей в SSE ([пример обсуждения](https://github.com/FlowiseAI/Flowise/issues/4877)) |
| F5 | При ответе **одним JSON** поток может выглядеть как **кусочки JSON** — нужна договорённость: отдельный текстовый стрим + мета в конце, или парсинг нарастающей строки, или стрим только после смены контракта | Иначе плохой UX |

## G. Инфраструктура и лимиты VPS

| # | Проверка | Критично |
|---|----------|----------|
| G1 | **Регион**: VPS, API LLM и (если облачный) эмбеддинг — без лишних кругов света | RTT доминирует в мелких запросах |
| G2 | **4 GB RAM / 2 vCPU**: норм для MVP; следить за суммой контейнеров; тяжёлый **локальный** эмбеддер + Qdrant + Flowise на одном хосте — типичное узкое место | При росте — вынести эмбеддинги в API или увеличить RAM |
| G3 | Rate limits провайдера модели и очереди при пиках | Защита от деградации «всем сразас медленно» |

## H. Оптимизации «ускорить без потери качества диалога» (приоритет)

Ниже — то, что чаще всего даёт **измеримый** выигрыш при сохранении качества, если сделано с A/B на реальных логах:

1. **Условный пропуск RAG** там, где контекст уже в state или воронка не про информацию — меньше шагов и токенов.  
2. **Ужать контекст в LLM**: topK после rerank, лимит символов на чанк в промпте, жёсткий cap на RAG-блок.  
3. **Короче история и промпт** без удаления обязательных правил (JSON, handoff, тон).  
4. **Быстрая или «лёгкая» модель** на простых ветках (если граф позволяет разделить) — только после метрик, иначе просадка по сложным кейсам.  
5. **Семантический кэш** для повторяющихся вопросов вне воронки — с порогом и инвалидацией KB.  
6. **Стриминг токенов в виджет** — в основном про **воспринимаемую** скорость; суммарное время генерации не обязано упасть.

Явный **non‑quick‑win**: «заменить Flowise на Python ради скорости» без смены пайплайна — **само по себе** почти не даёт выигрыша; выигрыш даёт **другая архитектура** (стрим, кэш, RAG).
