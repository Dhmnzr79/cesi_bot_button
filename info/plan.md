1. Файловая структура (Архитектура проекта)
Организуй файлы в Cursor следующим образом:

/knowledge_base — Твои .md файлы с медицинскими фактами. (Обязательно добавь метаданные в начало, например: topic: implants).

/config/conversion_map.json — Управляющий файл. Здесь мы храним «рельсы» диалога и кнопки.

/src/classifier_prompt.txt — Инструкция для «Нулевого этапа» (распознавание Да/Нет).

/src/main_prompt.txt — Основной системный промпт (логика ответов).

2. Слой данных: conversion_map.json (Пространство для маневра)
Этот файл позволяет тебе выбирать: хочешь ты кнопки или просто текст для конкретной темы.

JSON
{
  "implants-pain": {
    "topic_id": "pain",
    "marketing_hook": "У нас есть видео, как проходит процедура в седации.",
    "cta_text": "Хотите посмотреть видео или рассказать подробнее про анестезию?",
    "suggested_buttons": ["Посмотреть видео", "Про анестезию", "Цены"] 
  },
  "default": {
    "topic_id": "general",
    "marketing_hook": "Я могу помочь с любым вопросом по стоматологии.",
    "cta_text": "Что вас интересует в первую очередь?",
    "suggested_buttons": [] 
  }
}
Фишка: Если массив suggested_buttons пуст — бот просто задаст вопрос текстом. Если там есть строки — бот выведет их как кнопки. Ты управляешь этим без программирования!

3. Логика "Нулевого этапа" (Intent Classifier)
Этот блок в Cursor должен обрабатывать входящее сообщение до того, как оно попадет в основную модель.

Промпт для классификатора:

"Проанализируй сообщение пользователя и историю чата.

Если это новый вопрос — верни intent: new_question.

Если это короткое подтверждение (Да, ок, давай, +, нажал на кнопку подтверждения) — верни intent: confirmation.

Если это отказ (Нет, не надо) — верни intent: rejection."

4. Основной системный промпт (Двигатель диалога)
Это сердце системы. Скопируй эту логику в Cursor. Она автоматически адаптируется под наличие кнопок.

Plaintext
ТЫ — ЭМПАТИЧНЫЙ ЭКСПЕРТ-КОНСУЛЬТАНТ КЛИНИКИ.

ИНСТРУКЦИЯ ПО КОНТЕКСТУ:
1. Если {{intent}} == 'confirmation', НЕ ИЩИ новую информацию. Используй данные из последнего сообщения в {{chat_history}} и раскрой следующий аспект.
2. Используй факты только из предоставленных чанков: {{context}}.

ИНСТРУКЦИЯ ПО ОФОРМЛЕНИЮ (ГИБКОСТЬ):
1. Проверь наличие данных в {{conversion_map}} для текущей темы.
2. Если в поле 'suggested_buttons' есть элементы:
   - Сформулируй ответ.
   - В самом конце добавь кнопки в формате: [Название кнопки].
3. Если поле 'suggested_buttons' пустое:
   - Заверши ответ открытым вовлекающим вопросом, чтобы продолжить диалог текстом.

СТИЛЬ: Никакого официоза. Дружелюбно, кратко, экспертно.
5. Технический Pipeline (Как это работает в связке)
Прием сообщения: Cursor получает ввод от юзера.

Классификация: Определяем: это «Да» или «Новый вопрос»?

Поиск (RAG): * Если «Новый вопрос» — ищем по базе .md с фильтром по topic_id.

Если «Да» — пропускаем поиск, берем старые чанки из памяти.

Сборка: Скрипт берет кусок из conversion_map.json для этой темы.

Генерация: LLM видит и факты, и кнопки (если они есть), и историю. Она выдает ответ.

🛡 Что это дает тебе как не-программисту?
Защита от ошибок: Ты не сломаешь код, если решишь убрать кнопки. Просто сотри их в JSON.

Экономия времени: Тебе не нужно переписывать .md файлы при каждой акции.

Тестирование: Ты можешь запустить бота с кнопками в одной теме (например, "Цены") и без кнопок в другой ("Страх боли") и сравнить, где люди чаще оставляют контакты.

Техническая карта: "Hybrid Navigational AI"
1. Файловая структура
/knowledge_base: Все твои .md файлы. Название каждого: topic__type__subtopic.md (например, implants__faq__pain.md).

/config/conversion_map.json: Файл с «умными» настройками для тестовых тем.

/src/classifier_prompt.txt: Промпт для распознавания «Да/Ок».

/src/main_prompt.txt: Главный системный промпт.

2. Работа с данными (Два режима)
Система автоматически выбирает режим для каждого сообщения:

Режим А: "Smart" (по Карте Конверсии)

Применяется, если topic из имени файла найден в conversion_map.json.

Бот использует маркетинговые крючки и кнопки из JSON.

Режим Б: "Auto" (только Метаданные)

Применяется для всех остальных файлов.

Бот просто отвечает по тексту документа и задает обычный уточняющий вопрос.

3. Конфигурация conversion_map.json (Тестовый пример)
Создай этот файл для 2–3 тем, чтобы показать заказчику «магию».

JSON
{
  "implants-pain": {
    "natural_followups": ["анестезия", "длительность операции"],
    "video_url": "https://youtube.com/...",
    "final_cta": {
      "text": "Хотите записаться на консультацию?",
      "buttons": ["Записаться", "Рассчитать стоимость"]
    }
  },
  "default": {
    "natural_followups": [],
    "final_cta": {
      "text": "У вас остались еще вопросы?",
      "buttons": []
    }
  }
}
Для теста: Заполни implants-pain.

Для остальных: Блок default будет просто поддерживать вежливый диалог без лишних кнопок.

4. Инструкция для промпта в Cursor
Добавь эту логику в основной промпт, чтобы бот знал, как себя вести:

Проверка темы: Посмотри, есть ли текущая тема в conversion_map.

Если тема "тестовая": * Используй natural_followups для подсказок в тексте.

Показывай кнопки final_cta, если юзер сказал «Спасибо» или «Понятно».

Если темы нет в карте:

Просто отвечай на основе текста файла.

В конце спроси: «Что еще вас интересует по этой теме?».

5. Почему это идеально для демонстрации?
Вау-эффект: На горячих темах (имплантация) бот ведет себя как крутой менеджер: предлагает видео, выводит кнопки.

Надежность: На всех остальных темах бот ведет себя как стабильный эксперт: четко отвечает по базе, не делает ошибок.

Масштабируемость: Ты говоришь заказчику: «Видите, как круто работает ветка про боль? Чтобы так же работали остальные 20 тем, нам нужно просто дозаполнить этот JSON-файл».
