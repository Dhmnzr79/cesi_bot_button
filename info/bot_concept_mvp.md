# Концепция бота (MVP)

## 1) Идея бота с точки зрения маркетинга

Бот задуман как "умный лендинг в диалоге", а не как медицинский консультант.
Его задача не лечить и не вести долгие разборы, а:

- быстро дать человеку полезный и понятный ответ по теме запроса;
- снять базовую тревогу простым человеческим тоном;
- предложить логичный следующий шаг;
- довести до записи на консультацию или передать администратору.

Ключевая продуктовая логика:

- минимум лишней сложности;
- максимум понятности для обычного пользователя;
- короткие ответы вместо длинных лекций;
- одна цель: конверсия в лид без давления.

По сути это цифровой "первый контакт":
человек узнает базовую информацию, получает ориентиры и переходит к живому человеку (врач/администратор) для персональных решений.

## 2) Идея metadata и `subtopic`

Почему просто `topic` недостаточно:

- внутри одной темы (например, имплантация) есть много разных подтем:
  боль, сроки, стоимость, приживляемость, виды имплантов и т.д.;
- если держать только `topic`, бот начинает смешивать контекст и "прыгать" между разными файлами;
- пользователь получает общий ответ "про всё сразу", а не точный ответ по текущей линии разговора.

Почему нужен `subtopic`:

- `subtopic` фиксирует конкретный фокус диалога;
- помогает удерживать линию беседы;
- уменьшает нерелевантные подмешивания;
- позволяет раскрывать тему пошагово (и не дублироваться).

Роль metadata:

- в каждом документе базы есть `topic`, `type`, `subtopic`, `filename`;
- эти поля становятся навигацией для бота;
- бот понимает не просто "про имплантацию", а "про приживляемость имплантов" и продолжает именно это.

## 3) Наша реализация (AgentFlow v2, MVP)

### 3.1 Базовая схема потока

1. `Intent Classifier LLM` определяет:
   - `new_question`
   - `confirmation`
   - `rejection`

2. `Router` + `IF` решают, нужен ли retrieval:
   - если `leadIntent = none` и `intent = new_question` -> retrieval включается;
   - иначе используется уже сохраненный контекст (`last_context`).

3. `Qdrant Retrieval` возвращает документы с metadata.

4. `Filter Retrieved Context` (Custom Function) сужает retrieved-документы по:
   - `meta_topic`
   - `meta_subtopic` (если уже известен),
   и сохраняет отфильтрованный результат в `last_context`.

5. `Load Conversion Map` подмешивает маркетинговые элементы по текущему контексту (если есть совпадение).

6. `Main LLM` формирует структурированный JSON-ответ.

7. `Parse + Validate + Save`:
   - валидирует поля ответа;
   - сохраняет state;
   - добавляет fallback-логику (например, для кнопок подтем);
   - инкрементирует `message_count`.

### 3.2 Минимальный state (7 полей)

- `intent`
- `leadIntent`
- `meta_topic`
- `meta_subtopic`
- `should_retrieve`
- `last_context`
- `message_count`

Это "минимально достаточно" для устойчивого поведения без овер-инжиниринга.

### 3.3 Логика ветвления по структуре контента

Ветвление строится не по "важности услуги", а по структуре файла:

- если в контексте один блок (`Коротко`) -> без расширенного диалога;
- если в контексте несколько подтем (`### ...`) -> бот предлагает продолжение по ним.

`followup_options`:

- формируются из реальных подтем текущего контекста;
- не выдумываются;
- в режиме записи всегда пустые.

### 3.4 Работа с "Да" и подтверждениями

Бот трактует согласие по смыслу последнего шага:

- если пользователь подтвердил продолжение темы -> раскрывает следующий subtopic;
- если подтвердил запись -> запускает шаги воронки (`awaiting_name` -> `awaiting_phone`).

Принцип UX:
"один ответ = один следующий шаг".
Это убирает зацикливание и двусмысленность.

### 3.5 Роль маркетинга в реализации

Marketing-слой встроен аккуратно:

- база знаний дает фактический ответ;
- conversion map добавляет релевантные крючки (видео, CTA, опции);
- бот не давит, а подводит к действию в нужный момент.

Итог:
бот остается полезным и "человечным", но технически контролируемым и предсказуемым.

---

## 4) Что считаем успехом MVP

MVP считается успешным, если:

- бот удерживает диалог на уровне `subtopic`;
- не дублирует подтемы и не выдумывает кнопки;
- корректно обрабатывает подтверждения;
- не уходит в медицинские консультации вне базы;
- стабильно доводит до записи или handoff администратору.
