# Архитектура поведения чат-бота ЦЭСИ (MVP, v1)

Нормативный документ: как устроены диалог, UI и слои ответственности. Соответствует стеку **Flowise AgentFlow** → **Parse** (`parse_validate_save.mvp.js`) → **виджет** (`widget.js`) и промпту **`prompt.mvp.md`**.

---

## 1) Поток данных (кратко)

1. **Start** → **Router** (Custom Function): выставляет `should_retrieve` в `flow.state` (учёт `leadIntent`, `situation_pending`, опционально `intent`).
2. **IF** → при `true` **Retriever (Qdrant)** → **Custom Function** (фильтр контекста, `last_context`) → **Main LLM** (structured JSON).
3. При `false` — обход RAG, тот же CF + LLM (контекст из прошлого хода при необходимости).
4. **Parse + Validate + Save**: валидирует JSON, карты CTA/VIDEO, сценарий «ситуация», whitelist followup по `###` документа, лимиты кнопок, пишет `flow.state`.
5. **Виджет** парсит финальный JSON, рисует текст, followup, кнопку видео (раскрытие по клику), «Рассказать о своей ситуации», CTA по матрице; фолбэк UI для шага «ситуация», если бэкенд ответил не по контракту.

Отдельного **Intent Classifier LLM** нет: один основной вызов LLM за ход с JSON-контрактом.

---

## 2) Навигация по теме (followup-кнопки)

- Кнопки ведут **только по структуре одного markdown-документа** (один файл в базе).
- **Подтема** = заголовок `### …` в этом файле, кроме **`### Коротко`** (не кнопка, не в навигации).
- **Single-topic** (только «Коротко» или одна суть без отдельных `###`) → followup-кнопок нет.
- **Запрещено**: выдуманные подписи; переходы на другие документы через кнопки.
- **Parse**: подписи **фильтруются** по фактическим `###` текущего документа из `last_context`; при несовпадении с LLM подставляются корректные заголовки (до **2** кнопок за ход).
- Поле ввода у пользователя **всегда** доступно.

### 2.1 Логика количества (продукт + Parse)

- **До 2 followup** в JSON после Parse.
- **Вместе с CTA** на экране — **не более 3** кнопок: followup(s) + опционально одна «конверсия» (видео **или** «ситуация», не оба в одном ответе при 1 followup) + CTA записи, либо только часть набора по матрице виджета.
- **2+ followup** в ответе → видео из карты **не** подставляется (Parse).

### 2.2 Порядок в UI (виджет)

Текст ответа → **followup** → **«Посмотреть видео с врачом»** (по клику плеер) → **«Рассказать о своей ситуации»** (если разрешено) → **CTA записи** (primary, внизу).

---

## 3) CTA (запись)

- Одна **primary**-кнопка: запись / контакт; текст **`cta_label`** задаёт **Parse** из `CTA_MAP`, не модель.
- Показ CTA на фронте — по **матрице виджета** (число followup, глубина диалога, прогрев); на шаге **ожидания текста «ситуация»** и при **handoff** по правилам виджета отдельные CTA не мешают вводу.
- Иконка календаря, отличие от вторичных кнопок — по дизайну виджета.

---

## 4) Конверсии (видео и «ситуация»)

**Scope v1:** только **видео** в чате и сценарий **«Рассказать о своей ситуации»**. Файлы, акции, квиз — вне scope.

### 4.1 Видео

- URL и заголовок из **`VIDEO_MAP`** внутри Parse (синхронизация с `config/video_map.json`).
- В UI — кнопка **«Посмотреть видео с врачом»**, воспроизведение **после клика**, не автоплеер в ленте.

### 4.2 «Рассказать о своей ситуации»

**Parse выставляет** `show_situation_button` при: `leadIntent === none`, `meta_topic === implantation`, не более **1** followup, оффер ещё не «израсходован» для пары `topic+subtopic` в state, пользователь не в `situation_pending`.

Если в том же ответе уже есть **видео из карты**, `show_situation_button` **снимается** (лимит 3 кнопок с CTA).

После клика: фиксированный текст приглашения и **«Назад к диалогу»** в JSON; `situation_pending`; Router отключает RAG на этот ход. Далее — свободный текст пользователя → сохранение в `situation_note`, затем переход к запросу имени (или выход «Назад»).

**Виджет** дополнительно держит флаг **`situationAwaitingNote`**: если бэкенд не прислал `situation_pending`/кнопку «Назад», UI всё равно показывает только «Назад» и не показывает запись/handoff на этом шаге.

#### 4.2.1 Ответ после ввода ситуации (маркетинг, тёплый лид)

Пользователь, описавший проблему (часто с телефона), — **тёплый лид**; ответ вида только «Спасибо, как вас зовут?» **недопустим**. Нужны **благодарность, признание контекста и коротко «что дальше»** перед вопросом имени.

**Источник текста на этом ходе — не Main LLM**, а **детерминированный сценарий** (шаблоны), чтобы:

- не добавлять задержку и нагрузку на основную модель;
- гарантировать стабильный тон и копирайт;
- легко править формулировки в одном месте.

**Где реализовать:** **Parse + Validate + Save** и/или отдельная **Custom Function** на ветке «принят валидный `situation_note`»; в финальном JSON подставляется готовое поле **`answer`**, затем **`leadIntent` → `awaiting_name`**. На этом ходе **обход Main LLM** (или короткое замыкание графа до Parse), иначе смысл теряется.

**Гибрид шаблонов (рекомендуется):** **2–4 готовых текста**, выбор по **простым правилам без LLM**, например:

| Условие | Шаблон |
|--------|--------|
| короткое/сухое сообщение | A — нейтрально-тёплый |
| есть маркеры тревоги («боюсь», «страшно», «переживаю», «не приживётся», «больно» и т.п.) | B — усиленная эмпатия |
| телефон уже собран в воронке | C — без давления на контакт, акцент на передаче комментария и имя |
| длинное личное сообщение | D (или вариант A) — усиленная благодарность за подробности |

Правила и списки слов — **в коде Parse/CF или в конфиге**, версионируются вместе с репозиторием.

**Примеры смысла (не дословная фиксация — копирайт правится в одном месте):**

- **A:** благодарность за описание → комментарий передадут в клинику для контекста → как к вам обращаться?
- **B:** признание волнения → комментарий поможет внимательнее подойти к случаю → как к вам обращаться?
- **C:** благодарность за ситуацию → для корректной передачи обращения уточнить имя (без повторного запроса телефона, если он уже в state).

Пустой ввод, флуд и «Назад» — по существующей логике Parse/Router, без подстановки шаблона приглашения к имени.

## 5) Тон ответа (два режима)

В продукте зафиксированы **ровно два** значения поля **`tone`** в JSON: **`default`** и **`supportive`**.

| Значение     | Когда (источник истины для поля) | Смысл подачи в тексте `answer` |
|--------------|-----------------------------------|---------------------------------|
| `supportive` | `meta_topic === implantation`     | Один короткий снимающий тревогу заход, дальше только факты из базы своими словами. |
| `default`    | Все остальные темы                | Сразу по сути, без обязательного «эмпатийного» вступления. |

- **Поле `tone` в JSON** детерминированно выравнивается в **Parse** по `meta_topic` (модель может указать то же; расхождение не критично).
- **Текст ответа** в обычных ходах не задаётся Parse: формулировки **обязана** соблюдать модель по **`prompt.mvp.md`**. **Исключение:** шаблоны после ввода «ситуации» (§4.2.1) — готовый **`answer`** подставляет Parse/CF. Отдельного кода «подмешивания тона» для обычных ответов нет.

Опционально в будущем: поле `tone` в metadata чанков Qdrant — **не** часть обязательного контракта v1.

---

## 6) Связки в конце ответа (навигационная и действийная)

**Связка** — одна короткая фраза в конце **`answer`**, без второго вопроса и без рекламного нажима. **Программно не вставляется**: только инструкции в **`prompt.mvp.md`**.

| Тип | Условие (логика продукта) | Цель |
|-----|---------------------------|------|
| **Навигационная** | В ответе есть **followup-кнопки** (продолжение по подтемам) | Мягко подвести к выбору следующего блока темы. |
| **Действенная** | **Нет** followup (или финал ветки), дальше логичен шаг к записи | Мягко подвести к следующему шагу (запись / консультация), не дублируя дословно текст CTA-кнопки. |

Ограничения: не больше **одной** такой фразы за сообщение; можно обойтись без неё, если смотрится лишней.

---

## 7) Источник фактов и пересказ

- Факты — **только** из контекста базы (RAG). Запрещено выдумывать цены, сроки, клинические обещания.
- В **`answer`** — **пересказ своими словами**, без копипасты markdown-структуры чанков; цифры и требования из базы сохранять точно.
- Подробности — в **`prompt.mvp.md`** (§3.1 и JSON-контракт).

---

## 8) Короткие ответы пользователя («да», «ок»)

- Интерпретация — по **последнему сообщению бота** и короткой истории; отдельный классификатор не используется.
- Предложили запись → «да» трактуем как согласие на воронку (`awaiting_name` и далее по сценарию).
- Предложили продолжить тему → «да» = продолжение по контексту.
- Неоднозначно → **один** короткий уточняющий вопрос или handoff при токсичности/флуде.

Правила простоты: в одном сообщении бота — **один** следующий шаг (не две вилки «и запишем, и расскажем» в одной реплике).

---

## 9) Handoff и чувствительные случаи

- Жалоба, негатив, сложные симптомы вне базы, явный запрос человека, флуд → **`meta_shouldHandoff: true`** (промпт + Parse не ослабляет это произвольно на шаге «ситуация» — см. Parse).
- На шаге ввода комментария «ситуация» handoff и запись на приём **не показываются** в виджете по правилам UX.

---

## 10) Связанные подтемы (конец документа, не по порядку)

Если пользователь попал сразу в «хвост» документа:

- Показать контент текущей подтемы.
- Предложить **1–2 оставшихся** заголовка `###` **этого же файла** (не семантический поиск, не другие документы).
- CTA и конверсии — по общим правилам лимита кнопок.
