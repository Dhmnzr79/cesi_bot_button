## Архитектура: диалог по **subtopic**, не только по **topic** (зафиксировано)

**Реализация в репозитории:** пошагово — **`info/agentflow_subtopic.md`** (Start, Qdrant, CF после ретривера, Load Conversion Map, Main LLM, Parse).

**Правильное понимание:** бот держит диалог по **SUBTOPIC**, а не только по **TOPIC**.

### Как это должно работать (сценарий)

1. **User:** «Больно ли ставить имплант?»
2. **Qdrant** находит, например, `implantation__faq__pain.md`.
3. **Сохраняем в state (ключевое):**
   - `meta_topic` = `implantation`
   - **`meta_subtopic` = `pain`** ← обязательно для удержания линии диалога.

4. **User:** «А долго?» (короткий вопрос без явной темы)
5. **Бот не теряет контекст:** retrieval **с фильтром** по уже известным полям:
   - `topic: implantation`
   - **`subtopic: pain`**
   - `query` = семантика текущей реплики (например «долго»).
6. Поиск **только** внутри релевантных чанков этой подтемы → ответ про длительность **в рамках той же линии** (например длительность операции в контексте «боль/процедура»), а не случайный чанк «приживление 3–6 месяцев» из другого подфокуса имплантации.

### Без subtopic (плохо)

- Фильтр только `topic: implantation` → широкий семантический поиск по **всем** файлам имплантации → риск нерелевантного чанка.

### С subtopic (правильно)

- Фильтр `topic` + **`subtopic`** → узкий поиск → ответы согласованы с **текущей подтемой** диалога.

### Что это значит для потока (на будущее, без реализации сейчас)

- В **`$flow.state`** нужен устойчивый **`meta_subtopic`** (объявление в Start + обновление после ретрива / Main LLM / Parse — по выбранной схеме).
- **Qdrant Retriever:** динамический **metadata filter** на `topic` + `subtopic` из state (и query = preprocess/clean question).
- **Conversion map:** ключи уже завязаны на связку topic/subtopic (как `implantation__pain` в `config/conversion_map.json`).
- **Классификатор / короткие реплики:** при `confirmation` и уточняющих вопросах **не сбрасывать** subtopic без явной смены темы.

## followup_options только при «есть куда вести» в источнике (зафиксировано)

**Проблема (пример):** ответ по `doctors__doctor__larin.md` с выдуманными «Расскажите о специализации врача» / «Как записаться» в `followup_options` и фразой «если хотите узнать подробнее…», хотя в файле **один** связный блок без нескольких явных `###`-секций для поэтапного раскрытия.

**Правило:**

- Расширенный диалог (кнопки / `followup_options` / явные «следующие шаги» по содержанию) — **только** если в **текущем контексте из базы** есть **несколько самостоятельных смысловых блоков**, по которым логично продолжить (типично — несколько заголовков `### …` с разным содержанием, как в `clinic__info__technology.md`: Коротко → Как планируется… → Как повышается точность…).
- Формулировки followup должны **соответствовать реальным разделам документа**, а не шаблонам «узнать подробнее» без якоря в тексте.
- Если документ **не** даёт отдельных несообщённых секций для следующего хода → **`followup_options: []`**, в `answer` **не** создавать видимость скрытого блока «про специализацию», если он уже покрыт одним ответом из чанка.

**К обсуждению (ничего не меняли в промптах):** как формализовать в `prompt.legacy.md` / Parse — отдельная задача после согласования.

## UX: «Да» после двойного предложения (рассказать / записаться) — к обсуждению

**Наблюдение (пример диалога):**

1. Бот в `answer` предлагает: *«…могу рассказать подробнее или помочь записаться на консультацию»*, плюс в `followup_options` узкие вопросы по теме (анестезия и т.д.).
2. Пользователь: **«Да»**.
3. Бот снова задаёт выбор: *«Вам удобнее сначала записаться… или подробнее разобрать…?»*, `followup_options` пустой — пользователю неудобно: согласие не продвигает сценарий, а возвращает к тому же развилку.

**Суть проблемы:** при **confirmation** на сообщение, где **явно** перечислены два пути (контент vs запись), одно слово «Да» не привязано к ветке — модель **переспрашивает** вместо того, чтобы применить **иерархию «Да»** из `prompt.legacy.md` (например: если в предыдущем ходе было и про запись, и про продолжение темы — выбрать **один** предсказуемый следующий шаг по правилам, а не дублировать вопрос).

**К обсуждению (без правок сейчас):**

- Жёстче зафиксировать в промпте: при «Да» после такого блока — **не** повторять тот же выбор; либо по умолчанию вести в **следующий информационный блок** по базе, либо в **запись** — по заранее выбранной приоритизации (см. «The Yes Rule» / CTA в `prompt.legacy.md`).
- Возможно связать с **последним явным приоритетом** в тексте бота (что шло первым/последним в предложении).
- Отдельно: не опустошать `followup_options` на втором шаге без замены на осмысленное продвижение.

## Правки MVP: `meta_confidence` и `flags_emotional` (к обсуждению)

**Идея:** для первой версии можно **не требовать** от модели эти поля в JSON — упростить Structured Output и Parse.

| Поле | Зачем было | MVP |
|------|------------|-----|
| **`meta_confidence`** | Тон, оговорки; в `about.md` — скромная роль, не для ветвления «доверия» | Убрать из схемы LLM **или** в **Parse** всегда подставлять константу (например `0.7`) под контракт виджета |
| **`flags_emotional`** | Явный флаг «тревога → не давить на запись» | Убрать из схемы **или** дефолт `false` в Parse; эмпатию оставить **только текстом** в промпте без отдельного поля |

**Условие:** согласовать с **контрактом ответа** (виджет / API): либо урезанный JSON, либо те же ключи с дефолтами на бэке/Parse.

**Когда вернуть поля:** аналитика, A/B, отдельная логика UI по флагу тревоги.
